\section{Goal and Scope}

The project asks whether a minimal agent can specialize itself for a class of
tasks. I instantiate the idea in a narrow, objectively scored domain: Python
bug repair on BugsInPy. The agent starts as a single-call direct bug fixer and
searches over a bounded configuration space: prompt constraints, reasoning
strategy, tool availability, examples, maximum iterations, and temperature.

The important design choice is that this is not a universal agent. It is a
stem agent for one task class: repairing localized Python bugs when given a
failing test output and candidate buggy line ranges. That scope is deliberately
smaller than SWE-bench-style issue resolution. It avoids synthetic benchmark
saturation while keeping the evaluation runnable in a one-day setting.

\section{Benchmark and Splits}

The benchmark is a frozen subset of 26 real BugsInPy bugs from \texttt{youtube-dl}
(bugs 1--26), stored in \texttt{benchmark/subset.json}. Each task contains the
project, bug id, category, difficulty, and changed-line count. The harness checks out the buggy
commit, copies the maintainer test files from the fixed commit, applies the
agent's JSON edits, and runs the BugsInPy test command under WSL.

The prompt intentionally uses a fault-localization oracle: snippets around the
reference patch's old-file line ranges are shown, but the fixed code is never
shown. This means the benchmark evaluates patch generation and local reasoning,
not full repository-scale bug localization. That limitation is explicit because
otherwise the evaluation would overclaim.

The split is fixed in \texttt{benchmark/splits.json}:

\begin{center}
\begin{tabular}{lcl}
\toprule
Split & Size & Role \\
\midrule
Vital signs & 4 & Regression-safety invariant during differentiation \\
Calibration & 6 & Sensor analysis only \\
Dev & 8 & Champion/challenger selection \\
Test & 8 & Final held-out evaluation \\
\bottomrule
\end{tabular}
\end{center}

Vital-sign tasks were selected from bugs solved by the vanilla direct baseline.
Calibration, dev, and test are disjoint and fixed before any stem-agent search.
The test split is small, so all percentages are reported with counts.

\section{Stem-Agent Mechanism}

The agent genome is the \texttt{AgentConfig} dataclass in \texttt{src/config.py}.
It captures the system prompt, few-shot examples, output format, enabled tools,
tool policy, approach (\texttt{direct}, \texttt{cot}, \texttt{react},
\texttt{plan\_execute}), maximum iterations, temperature, and model. The
validator rejects malformed configurations, so every mutation produces a
runnable agent or no change.

The differentiation loop is implemented in \texttt{src/differentiator.py}.
Each iteration samples one or two mutation operators from \texttt{src/mutator.py},
evaluates the challenger on dev plus vital signs, and promotes it only if dev
pass@1 improves by more than \texttt{min\_improvement}. The safeguard rejects a
challenger if it regresses the vital-sign split below a threshold. This is the
main engineering idea: specialization is allowed only when it does not break
simple behavior the base agent already had.

The vital-sign threshold is set to 0.75 (3/4 must pass). With only four vital
tasks, a threshold of 1.0 made the loop unviable: one fragile task
(\texttt{youtube-dl:6}) that the baseline solves but most prompt variants break
caused every mutation to be rejected. Tolerating one regression out of four is
the minimum slack needed for the search to explore.

The sensor runs once on calibration results. In the one-day pipeline I added a
deterministic \texttt{--heuristic} mode to avoid spending extra calls only for
diagnostics. The generated report is in \texttt{results/sensor/sensor\_report.json}.
For this subset it found calibration pass@1 of 1/6, with failures concentrated
in \texttt{missing\_check}, \texttt{wrong\_variable}, and \texttt{other}. It
suggested \texttt{add\_constraint}, \texttt{specialize\_for\_category}, and
\texttt{switch\_approach}. The report explicitly records
\texttt{reference\_fixes\_used=false}; no fixed BugsInPy code is used by the
sensor.

\section{Current Evidence}

\subsection*{Model selection}

The model was selected from two candidates on 25 BugsInPy bugs (bug 16 was
validated separately and added to the frozen subset afterwards):

\begin{center}
\begin{tabular}{lcc}
\toprule
Model & Solved & pass@1 \\
\midrule
\texttt{gpt-5.4-nano-2026-03-17} & 6/25 & 24.0\% \\
\texttt{gpt-5.4-mini-2026-03-17} & 7/25 & 28.0\% \\
\bottomrule
\end{tabular}
\end{center}

The selected model is \texttt{gpt-5.4-mini-2026-03-17}. This is not a
statistically strong model comparison; it is a pragmatic one-day choice because
the gap is small and the sample is small.

\subsection*{Baseline}

On the frozen 26-bug benchmark, the vanilla direct baseline solves 7/25 tasks
(28.0\% pass@1). On the 8-task test split specifically, it solves 1/8:
\texttt{youtube-dl:13}.

\subsection*{Differentiation and ablations}

The differentiation loop, ablations, and final evaluation are produced by:

\begin{verbatim}
python scripts/run_differentiation.py --max-iterations 10 \
    --vital-threshold 0.75
python scripts/run_ablations.py --max-iterations 10
python scripts/run_random_search.py --max-iterations 10
python scripts/run_final_eval.py --agents vanilla_direct stem_agent \
    random_search no_sensor no_safeguard
\end{verbatim}

\subsection*{Test-split results}

All six agents were evaluated on the held-out 8-task test split. Bootstrap 95\%
confidence intervals are computed over 10\,000 resamples.

\begin{center}
\begin{tabular}{lcccr}
\toprule
Agent & Solved & pass@1 & 95\% CI & Cost \\
\midrule
\texttt{vanilla\_direct} & 2/8 & 25.0\% & [0.0, 62.5] & \$0.003 \\
\texttt{stem\_agent} & 2/8 & 25.0\% & [0.0, 62.5] & \$0.302 \\
\texttt{hand\_tuned} & 1/8 & 12.5\% & [0.0, 37.5] & \$0.298 \\
\texttt{no\_safeguard} & 1/8 & 12.5\% & [0.0, 37.5] & \$0.311 \\
\texttt{no\_sensor} & 1/8 & 12.5\% & [0.0, 37.5] & \$0.308 \\
\texttt{random\_search} & 1/8 & 12.5\% & [0.0, 37.5] & \$0.305 \\
\bottomrule
\end{tabular}
\end{center}

McNemar's test against vanilla is $p=1.0$ for all agents. With only 8 test
tasks, no statistically significant difference can be established between any
pair of agents. The stem agent matches vanilla on count but solves a
\emph{different} set of bugs: vanilla solves \texttt{youtube-dl:13} and
\texttt{youtube-dl:25}; the stem agent solves \texttt{youtube-dl:13} and
\texttt{youtube-dl:10}. The CoT approach gained one bug and lost one. No
conclusion about relative performance can be drawn from this sample size.

\subsection*{Dev-set differentiation}

The differentiation loop tells a more detailed story:

\begin{center}
\begin{tabular}{lcccc}
\toprule
Variant & Dev score & Iters & Accepted & Converged \\
\midrule
\texttt{stem\_agent} & 0.250 & 9 & 1 & yes \\
\texttt{no\_safeguard} & 0.250 & 7 & 1 & yes \\
\texttt{random\_search} & 0.375 & 8 & 2 & yes \\
\texttt{no\_sensor} & 0.125 & 10 & 0 & no \\
\texttt{vanilla\_direct} & 0.125 & -- & -- & -- \\
\bottomrule
\end{tabular}
\end{center}

Two conclusions can be drawn from the dev results despite the small sample.
First, random search achieved the highest dev score (0.375) but dropped to 1/8
on test. This is expected overfitting: without a safeguard, the search exploits
dev-set noise. Second, the no\_sensor variant accepted zero mutations: all ten
were rejected by the safeguard. Without sensor guidance the mutations are too
noisy to pass the vital-sign gate, so the search makes no progress. The stem
agent and no\_safeguard variants both found the same \texttt{switch\_approach}
mutation (direct $\to$ CoT) independently, which is a positive signal that the
search is reproducible when sensor guidance is present.

\subsection*{Ablation interpretation}

The ablation matrix:

\begin{center}
\begin{tabular}{lll}
\toprule
Variant & What it removes & Observed effect \\
\midrule
\texttt{stem\_agent} & (full system) & Found CoT, dev 0.250 \\
\texttt{no\_sensor} & Sensor guidance & Stuck at baseline, 0 accepted \\
\texttt{no\_safeguard} & Vital-sign gate & Same result as full system \\
\texttt{random\_search} & Both & Overfit dev (0.375), dropped on test \\
\bottomrule
\end{tabular}
\end{center}

Conclusion: the sensor is the critical component. Without it, the search cannot
find a viable mutation within budget. The safeguard contributed less on this
benchmark because the winning mutation (CoT) happened to preserve vital-sign
performance. Whether the safeguard becomes critical on a larger, more diverse
benchmark where mutations are more destructive remains an open question that
this dataset is too small to answer.

\section{Failure Modes and Tradeoffs}

The first attempted benchmark was synthetic and was abandoned because standard
direct prompting solved most tasks. That would have made any before/after claim
uninformative. Moving to BugsInPy made the task harder and exposed more realistic
failures: invalid JSON edits, syntactically invalid patches, patches that apply
but preserve the wrong behavior, and failures where the localized line range is
not enough context.

A concrete early failure: the initial safeguard threshold of 1.0 (all four vital
tasks must pass) made the differentiation loop reject every mutation. One vital
task (\texttt{youtube-dl:6}) was fragile enough that any prompt variation broke
it, even when the mutation improved dev score. Lowering the threshold to 0.75
(tolerate one regression) unblocked the search. This illustrates a general
tension: the safeguard must be strict enough to prevent real regressions but
loose enough that the search can explore.

The main risk is complexity without measured gain. The implementation therefore
keeps the search space bounded and logs every accepted or rejected mutation. If
the stem agent does not beat \texttt{vanilla\_direct}, the correct conclusion is
not that self-specialization works; it is that this bounded search did not find
a better configuration within the budget.

The second limitation is the fault-localization oracle. Showing candidate line
ranges makes the task cheaper and isolates patch generation, but it also makes
the setting easier than full real-world bug repair. A stronger follow-up would
replace this with retrieval over the repository and score on a larger,
multi-project BugsInPy or SWE-bench Lite subset.

\section{What I Would Do Next}

With more time I would expand the benchmark to multiple BugsInPy projects
(\texttt{thefuck}, \texttt{black}, \texttt{cookiecutter}) to reach 60+ validated
bugs, which would make the test split large enough for meaningful bootstrap
confidence intervals. The current 8-task test split is a pilot; any percentage
on it has wide error bars.

I would also run the differentiation loop multiple times with different seeds to
measure variance, compare against \texttt{vanilla\_cot} and a hand-tuned
baseline, and add a BugsInPy-aware verification tool for candidate JSON edits.
The compile step is a bottleneck: the BugsInPy framework installs dependencies
one-by-one and recreates the virtual environment from scratch for every bug.
Caching the virtual environment per project would cut validation time
substantially.
