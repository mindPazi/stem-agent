\section{Abstract}
\noindent
A \emph{stem agent} starts from a minimal, generic configuration and 
specializes itself through guided evolutionary search: it mutates its own 
prompt, tools, and strategy, measures each change against a held-out 
performance signal, and rejects changes that regress previously-solved tasks 
(\emph{vital signs invariant}). We instantiate this idea on Python bug repair, a domain with objective evaluation, and compare against vanilla baselines, 
a hand-tuned agent, and random mutation search. Existing frameworks (DSPy, 
OPRO, Promptbreeder, ADAS) optimize prompts or agent code but none formalize 
a regression-safety mechanism; this is our primary contribution.
\section{Benchmark and Experimental Setup}

\subsection{Benchmark}

The benchmark comprises \textbf{90 self-contained Python bug-repair tasks}.
Each task is a directory containing: \texttt{buggy.py} (the file with the injected bug), \texttt{fixed.py} (the ground-truth reference fix, never shown to agents), \texttt{test\_suite.py} (a \texttt{pytest} suite that fails on the buggy file and passes on the fixed file), and \texttt{metadata.json} (category, difficulty, source, description).

\textbf{Source.}
The initial benchmark was constructed via \emph{mutation testing} on Exercism Python exercises: correct reference solutions were taken and a single-point bug was injected using known mutation operators (off-by-one boundary shift, wrong relational operator, inverted boolean condition, wrong variable reference, missing guard clause, wrong API argument, boundary index error).
All tasks carry \texttt{"source": "mutation\_exercism"} in their metadata.

After a first model-calibration run, this benchmark proved too easy for current small OpenAI models (Table~\ref{tab:model-calibration}).
We therefore installed and smoke-tested BugsInPy~\cite{widyasari2020}, a dataset of real bugs from popular Python projects, as the candidate harder external dataset.
The BugsInPy framework checked out \texttt{youtube-dl} bug 2 successfully; the buggy version failed the provided test (\texttt{expected length 7, got 6}) and the fixed version passed the same test.

\textbf{Validation.}
Every task was validated automatically: \texttt{buggy.py} was run as the candidate solution and \emph{must fail} at least one test; \texttt{fixed.py} was run as the candidate solution and \emph{must pass} all tests.
Tasks that did not satisfy this invariant because the bugs were too subtle for the provided tests to catch were redesigned with targeted test cases that expose the fault; this is a correctness criterion independent of the leakage issue described below.
All 90 tasks passed validation before being included.

\textbf{Test-source leakage and prompt design.}
A manual inspection of 10 randomly sampled tasks during construction revealed a
test-leakage failure mode: in the originally generated test suites, ultra-specific
assertions of the form \texttt{assert f(x) == hardcoded\_value} produced
tracebacks that effectively spelled out the correct return value, and several
tests defined a local \texttt{\_ref} function that re-implemented the intended
algorithm. Both patterns directly compress the dynamic range of any agent
comparison, since a competent LLM can solve the task by pattern-matching the
test source rather than by reasoning about the buggy code. This is the textbook
``synthetic data too easy'' failure mode.

We address this in two ways. First, 12 tasks (\texttt{task\_004}, \texttt{task\_005},
\texttt{task\_008}, \texttt{task\_011}, \texttt{task\_012}, \texttt{task\_015},
\texttt{task\_021}, \texttt{task\_027}, \texttt{task\_042}, \texttt{task\_055},
\texttt{task\_057}, \texttt{task\_058}) where the original tests most directly
revealed the fix were rewritten with hardcoded behavioral assertions and minimal
spoilers (a separate concern from the correctness-driven redesigns in the
Validation step above); the remaining 78 retain behavioral or property-based assertions from
the initial generation, several of which still embed reference implementations
in helper functions. The full rewrite of the 12 tasks is reproducible via
\texttt{benchmark/fix\_tasks.py}.

Second, and more importantly, the agent prompt does \emph{not} include the test
source. The agent receives only \texttt{buggy.py}; the test suite is never shown
verbatim. Tests remain the binary oracle for scoring (pass/fail) and are
indirectly accessible to ReAct-style agents through the \texttt{run\_tests} and
\texttt{get\_traceback} tools, which return the failing pytest output but not the
test source code. This design choice both removes the leakage from any
unrewritten \texttt{\_ref} helpers and matches a more realistic bug-fixing
scenario, where a developer typically has a buggy file plus a failure report
rather than the full test suite verbatim. A regression test in
\texttt{tests/test\_agent.py} (\texttt{test\_test\_suite\_code\_never\_in\_prompt})
locks in this invariant.

We re-validated all 90 tasks after these changes: \texttt{buggy.py} fails at
least one test and \texttt{fixed.py} passes every test on each task.

\textbf{Model/dataset calibration.}
A first calibration of \texttt{vanilla\_direct} on all 90 synthetic tasks with three candidate small models reported pass@1 of 85.6\%, 86.7\%, and 92.2\% for \texttt{gpt-4o-mini-2024-07-18}, \texttt{gpt-5.4-nano-2026-03-17}, and \texttt{gpt-5.4-mini-2026-03-17} respectively.
Even the weakest candidate solved more than 85\% of the tasks with the minimal direct prompt, which left no usable headroom for a before/after differentiation claim.
We therefore moved the calibration onto a harder, real-bug subset and re-evaluated the candidate models there.

We took 25 contiguous \texttt{youtube-dl} bugs from BugsInPy and ran \texttt{vanilla\_direct} on each, prompting the model with the failing test output, the changed-file snippets, and the candidate buggy line ranges from the reference patch (no fixed code is shown).
Validation, calibration, and per-task evaluation are implemented in \texttt{scripts/run\_bugsinpy\_model\_calibration.py}; results land in \texttt{results/bugsinpy\_ytdl25\_model\_calibration/}.
We treat this as a model-selection probe, not as the project benchmark: the sample is small and single-project, and so are the conclusions we draw from it.

\textbf{Selected model: \texttt{gpt-5.4-mini-2026-03-17}.}
5.4-mini outperforms 5.4-nano on the same 25-bug subset (28.0\% vs 24.0\%) and lands in a range that leaves usable headroom for differentiation without saturating the benchmark.
We did not run a third full calibration of \texttt{gpt-4o-mini} on the BugsInPy subset: it is from an older family, was already the weakest of the three on the saturated synthetic probe, and re-running it would have consumed calibration budget without a plausible path to selection.
We acknowledge this leaves \texttt{gpt-4o-mini} as an inferred-rather-than-measured comparison on the BugsInPy subset and treat the choice between the two 5.4 candidates as the load-bearing decision.
A rigorous selection would involve statistical testing to verify whether the observed gap is above noise rather than a sampling artefact; with $n{=}25$ the difference is not significant on its own.
Given time constraints, and given the convergence of evidence (5.4-mini is a newer, larger model in the same family, it scored higher on both the saturated synthetic probe and the real-bug subset, and 4o-mini was already eliminated), we accept 5.4-mini as the working model without a formal significance gate.
The selected model is pinned in \texttt{src/config.py} (\texttt{DEFAULT\_MODEL}) and \texttt{configs/default.yaml}, and the \texttt{LLMClient} default uses the same identifier.

\textbf{Acknowledged benchmark limitations.} The benchmark is synthetic: every
task is a single-point mutation injected into a short reference solution from
Exercism. Difficulty skews easy, with 66 of 90 tasks (73\%) labeled
\texttt{easy} (single-token or single-operator change) and 24 (27\%) labeled
\texttt{medium}. The functions are short (typically under 20 lines), bugs are
isolated (no cross-function or cross-file effects), and the dataset does not
exercise repository-level reasoning, multi-hunk patches, dependency conflicts,
or subtle bugs requiring long-context understanding. We treat the benchmark as a
controlled testbed for the stem-agent mechanism rather than as a claim about
real-world bug-repair performance, and we report all results with this
constraint in mind. Section~\ref{sec:limitations} discusses this further.

\subsection{Splits and Evaluation Protocol}
\label{sec:splits}

The 90 tasks are partitioned into four non-overlapping splits, assigned once after the baseline run and frozen for the remainder of the project in \texttt{benchmark/splits.json}.

\begin{table}[h]
\centering
\caption{Split sizes and roles.}
\label{tab:splits}
\begin{tabular}{lcl}
\toprule
Split & Size & Used by \\
\midrule
Vital Signs & 8  & Phase 2 (regression-safety invariant) \\
Calibration & 12 & Phase 1 (sensor analysis only) \\
Dev         & 30 & Phase 2 (champion/challenger selection) \\
Test        & 40 & Phase 3 (final evaluation, never touched during search) \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Assignment procedure (deterministic, seed=42).}
After the baseline run, vital signs are sampled from the set of tasks that \texttt{vanilla\_direct} solves correctly, stratified by bug category so each category contributes proportionally.
The remaining tasks are then drawn into calibration, dev, and test in that order, again stratified by category.
Sampling is implemented in \texttt{src/splits.py} and the assignment is invoked by \texttt{scripts/assign\_splits.py}; the resulting JSON file is overwrite-protected by the script unless \texttt{--force} is passed.

\textbf{Freeze policy.}
The test split is never read during Phase 1 or Phase 2; only \texttt{scripts/run\_phase3.py} (Day 3) loads it.
A regression test (\texttt{tests/test\_splits.py}) verifies that the four splits are non-overlapping and exhaustive.

\textbf{Primary metric: pass@1.}
Per task, the agent emits a single fix; we run the full \texttt{pytest} suite in a sandboxed subprocess with a 30-second timeout and score the task as 1 if all tests pass and 0 otherwise.
Aggregate pass@1 is the mean over the test split.
We report counts (e.g.\ ``23/40'') alongside percentages so that the small split size remains visible.

\textbf{Statistical reporting (Day 3).}
Final pass@1 numbers will be reported with bootstrap 95\% confidence intervals (10{,}000 resamples of the test set) and pairwise McNemar's test against \texttt{vanilla\_direct}, alongside per-category breakdowns.

\textbf{Baselines.}
Five baselines establish the comparison frame: \texttt{vanilla\_direct} (minimal prompt, no tools, single LLM call); \texttt{vanilla\_cot} (same prompt with a chain-of-thought instruction); \texttt{generic\_react} (all seven tools enabled, generic prompt, ReAct loop with up to five iterations); \texttt{hand\_tuned} (a manually-engineered system prompt with a curated tool subset, intended as the realistic ceiling for human prompt engineering); and \texttt{random\_search} (the same mutation budget as the stem agent but without sensor guidance and without the vital-signs safeguard).
The four LLM-driven baselines are defined in \texttt{src/baselines.py} and run by \texttt{scripts/run\_baselines.py}; \texttt{random\_search} is produced by Day 3.

\textbf{Ablations.}
Four ablation variants of the stem agent isolate the contribution of each component: \texttt{no\_sensor} (Phase 1 skipped, mutations sampled uniformly from iteration 0), \texttt{no\_safeguard} (vital-signs check disabled, challenger promotion based on dev score alone), \texttt{no\_tool\_mutation} (only prompt and strategy operators active), and \texttt{no\_prompt\_mutation} (only tool, strategy, and inference operators active).

\textbf{Cost tracking.}
Every LLM call passes through \texttt{src/llm\_client.py}, which logs prompt and completion tokens, latency, and a per-call USD estimate to \texttt{results/api\_calls.jsonl}. Pricing for the 5.4 series is not published, so the cost estimate uses the \texttt{gpt-4o-mini} tier as a documented proxy: numbers are comparable across runs but should not be read as actual billing.
Total expenditure per phase is reported alongside the metrics.

\section{The Stem Agent}
\label{sec:stem-agent}

The agent's ``genome'' is a single \texttt{AgentConfig} dataclass (\texttt{src/config.py}) covering four independent axes:
prompt (\texttt{system\_prompt}, \texttt{few\_shot\_examples}, \texttt{reasoning\_instruction}, \texttt{output\_format});
tools (\texttt{enabled\_tools} drawn from a fixed registry of seven, \texttt{tool\_use\_strategy});
strategy (\texttt{approach} $\in$ \{\texttt{direct}, \texttt{cot}, \texttt{react}, \texttt{plan\_execute}\}, \texttt{max\_iterations});
and inference (\texttt{temperature}, \texttt{model}).
A validator runs at construction time and rejects out-of-range values, unknown tools, and incompatible combinations, so that every config produced by mutation is structurally well-formed.

\subsection{Baseline Establishment}
\label{sec:baselines}

Phase 0 establishes the performance floor and produces the data needed to assign splits.
The vanilla agent is defined as a single LLM call (\texttt{approach="direct"}) with the system prompt ``You are a Python developer. Fix the bug.'', no few-shot examples, no enabled tools, and \texttt{temperature=0.0} on the calibrated experiment model; this is the simplest possible configuration the rest of the system mutates away from.

The four configured baselines (\texttt{vanilla\_direct}, \texttt{vanilla\_cot}, \texttt{generic\_react}, \texttt{hand\_tuned}; Section~\ref{sec:splits}) are all run on the full set of 90 tasks via \texttt{scripts/run\_baselines.py}.
Per-task pass/fail is serialised together with the agent's fix, the test duration, and the cost of the call to \texttt{results/baselines/<name>.json}.

The vanilla pass/fail vector is then consumed by \texttt{scripts/assign\_splits.py} to produce \texttt{benchmark/splits.json} (Section~\ref{sec:splits}).
This is the only point at which the test split is touched before Phase 3: the per-task results from Phase 0 are stored, but no result that informs Phase 1 or Phase 2 reads from the test partition.

\subsection{Sensing}
\label{sec:sensing}

Phase 1 (\texttt{src/sensor.py}, run via \texttt{scripts/run\_sensor.py}) is a single, static diagnostic pass over the calibration split.
It reads the vanilla pass/fail labels and proposed fixes already produced in Phase 0, and emits a \texttt{SensorReport} consumed by the differentiation loop.
Importantly, the sensor and the \texttt{add\_exemplar} mutation operator are the only components allowed to inspect the reference \texttt{fixed.py} files, and only for calibration tasks; this is recorded explicitly in the report's \texttt{analysis\_scope} and \texttt{reference\_fixes\_used} fields so that the leakage scope is auditable.

Concretely, the sensor performs three steps.
First, it computes per-category counts and per-category pass rate on calibration.
Second, for each calibration task that vanilla \emph{failed}, it issues a single LLM call passing the buggy code, the agent's wrong fix, and the reference fix, and asks for a one-sentence concrete failure pattern (e.g.\ ``agent ignored boundary condition'' or ``agent changed the wrong variable'').
Third, it issues two aggregation calls: one to extract success patterns from passing tasks, and one to map the observed failure patterns and worst-performing categories onto a subset of the twelve mutation operator names listed in Table~\ref{tab:operators}.
The resulting suggestions are validated against the operator registry and any unknown names are discarded; if the LLM call fails or returns malformed JSON, a heuristic fallback (\texttt{specialize\_for\_category}, \texttt{add\_constraint}, plus \texttt{add\_exemplar} when boundary-class categories dominate) keeps the loop running.
The sensor never re-runs during Phase 2: its output is a frozen \texttt{results/sensor/sensor\_report.json}.

\subsection{Differentiation}
\label{sec:differentiation}

Phase 2 (\texttt{src/differentiator.py}) implements an evolutionary mutation-and-selection loop with a hard regression-safety invariant.
The loop runs for at most 20 iterations and stops early on convergence.

\textbf{Mutation operators.}
Twelve typed operators (Table~\ref{tab:operators}) act on the four genome axes.
Each operator takes an \texttt{AgentConfig} together with a \texttt{MutationContext} (sensor report, iteration index, mutation history, the pool of vanilla-solved calibration tasks, the LLM client, and a seeded \texttt{random.Random}) and returns a new, validated \texttt{AgentConfig}.
Three operators (\texttt{add\_exemplar}, \texttt{rephrase\_system}, and the optional reasoning trace inside \texttt{add\_exemplar}) are LLM-assisted; the remaining nine are deterministic given the RNG state.
Operators that would produce no change for the current config (e.g.\ \texttt{remove\_exemplar} when none are present, \texttt{enable\_tool} when all seven are already enabled) are filtered before sampling.

\begin{table}[h]
\centering
\caption{Mutation operators (\texttt{src/mutator.py}).}
\label{tab:operators}
\begin{tabular}{lll}
\toprule
Axis & Operator & Effect \\
\midrule
Prompt   & \texttt{add\_exemplar}            & Append a few-shot example from a solved calibration task \\
Prompt   & \texttt{remove\_exemplar}         & Drop a randomly-chosen few-shot example \\
Prompt   & \texttt{rephrase\_system}         & LLM-rewrite the system prompt preserving its semantics \\
Prompt   & \texttt{add\_constraint}          & Inject a debugging-rule sentence not already present \\
Prompt   & \texttt{remove\_constraint}       & Remove a previously-injected constraint sentence \\
Prompt   & \texttt{specialize\_for\_category} & Append category-specific guidance (worst-performing first) \\
Tools    & \texttt{enable\_tool}             & Add a tool to \texttt{enabled\_tools} \\
Tools    & \texttt{disable\_tool}            & Remove a tool from \texttt{enabled\_tools} \\
Strategy & \texttt{switch\_approach}         & Cycle direct $\to$ cot $\to$ react $\to$ plan\_execute \\
Strategy & \texttt{adjust\_iterations}       & Change \texttt{max\_iterations} by $\pm 1$ or $\pm 2$ (clamped to $[1,10]$) \\
Output   & \texttt{switch\_output\_format}   & Cycle full\_file $\to$ diff $\to$ function\_only \\
Inference & \texttt{adjust\_temperature}     & Nudge temperature by $\pm 0.1$ (clamped to $[0,1]$) \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Sampling strategy.}
Each iteration samples one or two operators (with weights $0.7$ and $0.3$) from the set of \emph{applicable} operators.
The sampling distribution is iteration-dependent.
For iterations $0$--$5$, when the sensor signal is most relevant, $70\%$ of picks come from the operators the sensor suggested and $30\%$ from the rest of the registry.
For iterations $6$--$15$ the sampler switches to uniform random over applicable operators, broadening the search.
For iterations $16$ and beyond, sampling is targeted: the worst-performing categories from the sensor report drive a preference for \texttt{specialize\_for\_category}, \texttt{add\_constraint}, and \texttt{add\_exemplar}.
An \emph{anti-oscillation} filter excludes any operator whose name appears in a rejected record within the last three iterations, unless removing it would empty the candidate pool.

\textbf{Champion / challenger selection.}
The challenger is evaluated on the union of dev and vital splits in a single batch, and the dev-set pass@1 is the score that drives selection.
A challenger is promoted to champion only when its dev score exceeds the current champion's by more than \texttt{min\_improvement = 0.02} (a 2 percentage point floor that prevents accepting noise on a 30-task split).
The champion configuration is saved to disk on every promotion (\texttt{champion.yaml}), so a crashed run can be inspected at the last accepted state.

\textbf{Vital-signs safeguard.}
Independently of the dev score, the safeguard (\texttt{src/safeguard.py}) requires that the challenger's pass rate on the vital-signs split satisfy $\ge \texttt{vital\_threshold}$ (default $1.0$).
A challenger that fails any vital task is rejected outright, and the iteration is logged with \texttt{reason="safeguard"}.
Unlike pure dev-score selection, this prevents the well-known failure mode where a specialized configuration trades easy-task correctness for marginal gains on harder ones.
The threshold is configurable on the command line so that the no\_safeguard ablation can be produced by setting it to $0$.

\textbf{Convergence.}
A separate check (\texttt{src/convergence.py}) terminates the loop early if no \emph{accepted} record in the last $5$ iterations had \texttt{delta} above \texttt{min\_improvement}.
This is conservative: rejected iterations alone never trigger early stopping, only a sustained absence of real improvements does.

\textbf{Logging.}
Every iteration appends one JSON object to \texttt{results/differentiation/differenti\-ation\_log.jsonl} containing the iteration index, the operators tried, the score before and after, the delta, the reason for the decision, and a snapshot of the challenger config.
A separate \texttt{champion\_history.json} records every accepted champion, and \texttt{differentiation\_summary.json} records the aggregate counts (accepted, safeguard rejections, no-improvement rejections) and the total API cost.
This is sufficient for the Day 4 mutation-effectiveness analysis to be derived purely from logs.

\textbf{Dry-run.}
\texttt{scripts/run\_differentiation.py --dry-run} replaces real evaluation with a deterministic mock (vital signs pass at $90\%$, dev tasks pass uniformly at $50\%$) and disables the LLM-assisted mutations.
This was used during Day 2 development to verify that all four iteration outcomes (acceptance, safeguard rejection, no-improvement rejection, convergence) are reachable without spending tokens.
