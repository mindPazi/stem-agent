\section{Goal and Scope}

The project asks whether a minimal agent can specialize itself for a class of
tasks. I instantiate the idea in Python bug repair on BugsInPy~\cite{widyasari2020}.
The agent starts as a single-call direct bug fixer and searches over a bounded
configuration space (prompt constraints, reasoning strategy, tool availability,
examples, temperature). The scope is deliberately narrower than SWE-bench~\cite{jimenez2024swebench}:
repairing localized Python bugs given a failing test output and candidate buggy
line ranges.

\section{Benchmark and Splits}

The first iteration used a synthetic benchmark of 90 tasks built via mutation
testing on Exercism Python exercises (single-point bugs: off-by-one, wrong
operator, missing guard, wrong variable). After running the vanilla baseline,
\texttt{gpt-4o-mini} solved ${\sim}80\%$ with a single direct call. This left
no measurable room for the stem-agent search to improve. A useful benchmark
requires the base model to sit in a middle range, not near 0\% and not near
100\%. I discarded the synthetic set and switched to BugsInPy~\cite{widyasari2020},
where the same model solves roughly 25--30\% under oracle fault-localization.

The benchmark is a frozen subset of 26 real BugsInPy bugs from
\texttt{youtube-dl}, stored in \texttt{benchmark/subset.json}. The prompt uses
a \textbf{fault-localization oracle}: snippets around the reference patch's
line ranges are shown, but the fixed code is never shown. This isolates patch
generation from bug localization; quantifying the oracle's advantage would
require a parallel full-file evaluation, left for future work.

This is a pilot subset, not the benchmark I would use for a final evaluation. The
intended workflow was to collect a larger set of BugsInPy bugs, validate each
bug locally to make sure the checkout, dependencies, and test command were not
broken by environment drift, and then execute the agent comparison on the
validated set. The time budget was enough to build and run the pipeline on this
small subset, but not to expand and validate the benchmark properly.

The split is fixed in \texttt{benchmark/splits.json}:

\begin{center}\small
\begin{tabular}{lcl}
\toprule
Split & Size & Role \\
\midrule
Vital signs & 4 & Regression-safety invariant \\
Calibration & 6 & Sensor analysis only \\
Dev & 8 & Champion/challenger selection \\
Test & 8 & Final held-out evaluation \\
\bottomrule
\end{tabular}
\end{center}

\textbf{Statistical power.} With $n{=}8$ binary outcomes, a one-sided exact
binomial test at $\alpha{=}0.05$ against $p_0{=}0.25$ requires $\geq 5/8$
successes to reject $H_0$; only a $\geq$37.5\,pp improvement is detectable.
The numbers below should therefore be read as a pipeline check and qualitative
failure analysis, not as evidence about the real task distribution.

\section{Stem-Agent Mechanism}

The agent genome is the \texttt{AgentConfig} dataclass: system prompt, few-shot
examples, output format, enabled tools, approach (\texttt{direct}/\texttt{cot}/\texttt{react}/\texttt{plan\_execute}),
max iterations, and temperature. The validator rejects malformed configs.

The \textbf{mutator} implements 15 operators in two tiers:
\emph{surface-level}~(12): prompt rephrasing, constraint injection, category
specialization, few-shot management, tool toggle, approach/format cycling,
temperature/iteration adjustment;
\emph{structural}~(3): self-verification (mentally re-run the test on the fix),
reflect-and-revise (second-pass self-critique), error-analysis protocol
(structured root-cause classification).
The space is deliberately bounded to serializable config changes. Every point
is reproducible and inspectable, answering a focused question: \emph{can
sensor-guided search beat random search within this space?}

The \textbf{differentiation loop} samples 1--2 mutations per iteration,
evaluates the challenger on dev + vital signs, and promotes only if dev~pass@1
improves by $>$\texttt{min\_improvement}. The \textbf{safeguard} rejects
challengers that regress vital signs below a threshold~(0.75). The
\textbf{sensor} runs once on calibration results, extracting failure patterns
and suggesting mutation directions. The sensor uses reference fixes to analyze
failure patterns in calibration tasks but never exposes them to the inner agent
(\texttt{reference\_fixes\_used=true} in \texttt{SensorReport}, scoped to
calibration only).
Anti-leakage: a unit test verifies that test-suite source code never appears in
the agent prompt.

\textit{Note:} In the 9-iteration run only 8 of 15 operators were sampled
(specialize\_for\_category, add\_constraint, switch\_approach, adjust\_temperature,
remove\_constraint, rephrase\_system, enable\_tool + add\_exemplar). The three
structural operators were not sampled; with more iterations or seeds, coverage
would increase.

\section{Current Evidence}

\subsection*{Model selection and baseline}

\texttt{gpt-5.4-mini-2026-03-17} was selected over \texttt{gpt-5.4-nano}
(7/25 vs.\ 6/25 on the full benchmark). On the 8-task test split, the vanilla
direct baseline solves 2/8.

\subsection*{Test-split results}

\begin{center}\small
\begin{tabular}{lccc}
\toprule
Agent & Solved & pass@1 & 95\% CI \\
\midrule
\texttt{vanilla\_direct} & 2/8 & 25.0\% & [0.0, 62.5] \\
\texttt{stem\_agent} & 2/8 & 25.0\% & [0.0, 62.5] \\
\texttt{hand\_tuned} & 1/8 & 12.5\% & [0.0, 37.5] \\
\texttt{no\_safeguard} & 1/8 & 12.5\% & [0.0, 37.5] \\
\texttt{no\_sensor} & 1/8 & 12.5\% & [0.0, 37.5] \\
\texttt{random\_search} & 1/8 & 12.5\% & [0.0, 37.5] \\
\bottomrule
\end{tabular}
\end{center}

All single-call agents (direct, CoT) cost $\sim$\$0.003.
\texttt{hand\_tuned} uses ReAct with tool calls, requiring $\sim$100$\times$ more
tokens for the same tasks with no accuracy gain in this pilot run. That is a
cost warning for this setup; it does not settle multi-step tool use in general.

McNemar $p{=}1.0$ for all pairs. The stem agent matches vanilla at 2/8 but
solves a \emph{different} pair (\texttt{youtube-dl:10} instead of
\texttt{:25}), suggesting that the CoT mutation changed the failure-mode
profile rather than aggregate accuracy. On a larger benchmark this shift might
translate to an improvement, or confirm that the two strategies trade bugs
rather than add them.

\subsection*{Dev-set differentiation}

\begin{center}\small
\begin{tabular}{lcccc}
\toprule
Variant & Dev score & Iters & Accepted & Converged \\
\midrule
\texttt{stem\_agent} & 0.250 & 9 & 1 & yes \\
\texttt{no\_safeguard} & 0.250 & 7 & 1 & yes \\
\texttt{random\_search} & 0.375 & 8 & 2 & yes \\
\texttt{no\_sensor} & 0.125 & 10 & 0 & no \\
\texttt{vanilla\_direct} & 0.125 & -- & -- & -- \\
\bottomrule
\end{tabular}
\end{center}

Two run-level observations (both from a \textbf{single seed}; see caveat below):
(1)~Random search achieved the highest dev score but dropped on test, a pattern
compatible with chasing noise on the tiny dev split.
(2)~Without sensor guidance, zero mutations were accepted; within this seed, the
search made no progress.
Both stem\_agent and no\_safeguard independently found \texttt{switch\_approach}
(direct$\to$CoT).

\subsection*{Ablation matrix}

\begin{center}\small
\begin{tabular}{lll}
\toprule
Variant & Removes & Observed effect \\
\midrule
\texttt{stem\_agent} & (full system) & Found CoT, dev 0.250 \\
\texttt{no\_sensor} & Sensor & No accepted mutations \\
\texttt{no\_safeguard} & Safeguard & Same as full system \\
\texttt{random\_search} & Both & Higher dev, lower test \\
\bottomrule
\end{tabular}
\end{center}

In this run, the sensor was the component associated with an accepted mutation.
The safeguard contributed less because CoT happened to preserve vital signs; a
larger validated benchmark would be needed to say whether that pattern holds.

\textbf{Single-seed caveat.} All runs use seed=42 with 10 iterations.
The difference between 0 and 1 accepted mutations could be sampling noise.
Running 3--5 seeds would show whether the same pattern repeats; this was not
feasible in the one-day budget.

\section{What Surprised Me}

\textbf{Pipeline cost.} I expected the benchmark pipeline to take less time.
Instead, validating BugsInPy tasks, handling local environment issues, and making
evaluation reproducible took most of the work. This is why the validated subset
stayed small.

\textbf{Small-model brittleness.} I expected \texttt{gpt-5.4-mini} to perform
better as a direct fixer on localized bugs. In the pilot, it solved only a small
fraction of tasks, even with fault-localized snippets. This matters for the
project because the whole setup is about whether configuration and prompting can
make a small agent more reliable.


\section{Limitations and Failure Modes}

\begin{enumerate}[nosep]
\item \textbf{No measurable improvement.} stem\_agent $=$ vanilla at 2/8.
  The search converges to CoT, which is the natural \texttt{vanilla\_cot}
  baseline. The most informative comparison, \texttt{vanilla\_cot} on the
  test split, was not run. The champion's \texttt{reasoning\_instruction}
  is effectively identical to vanilla\_cot's (same default fallback);
  only the system prompt differs (``expert \ldots\ Return only the corrected
  code''). Whether that prompt wording matters requires this comparison.
\item \textbf{Small validated subset.} The 26 validated bugs leave only 8 test
  tasks, making any difference below $\sim$37.5\,pp undetectable.
\item \textbf{Single seed.} All ablation observations rest on one run. The sensor's
  effect is only a run-level observation.
\item \textbf{Fault-localization oracle.} The agent sees candidate line ranges,
  making the task easier than real-world debugging.
\item \textbf{Bounded search space.} The 15 operators modify prompts and config
  only; no sub-agent composition, tool generation, or memory design.
\end{enumerate}

\section{What I Would Do Next}

\begin{itemize}[nosep]
\item The main next step is to expand the subset to 60+ validated bugs across
  multiple BugsInPy projects. I would first collect more bugs, validate each
  one locally to confirm that the buggy and fixed checkouts and tests still run,
  discard broken tasks, and only then run the agent comparison on the validated
  set.
\item After the benchmark is expanded, run 3--5 seeds to measure variance and
  check whether the sensor pattern repeats.
\item Run \texttt{vanilla\_cot} on the test split and compare directly.
\item Evaluate without the fault-localization oracle to quantify its advantage.
\end{itemize}
